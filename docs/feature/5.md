# Issue #5 — Weather condition/icon sensor

Branch: `feature/5_weather-condition-sensor`

## Problem

Expose the weather icon/condition per forecast day. Built on the #2 base entity,
mirroring #3/#4, with one twist: the value is a string enum (`iconUrl`), not a
number.

Requirements (from the ticket):
- Value from `DayData.icon_url` (e.g. `FullSun`, `LightRain`).
- Consider mapping the API icon strings to friendlier states and/or an `icon`.
- One entity per configured forecast day.

## Key constraint: the icon enumeration is undocumented

Probing the live API (all 5 days × 3 cultures) currently returns only `FullSun`
and `LightRain`, and the values are language-independent. But Burgers Zoo's full
set (cloudy, heavy rain, snow, fog, night variants, …) is unknown. **The design
must not break on a value we haven't seen.**

## Direction

**New `BurgersZooConditionSensor(BurgersZooBaseEntity)`** in `sensor.py`:
- `key = "condition"` → unique id `{entry_id}_condition_{day}`.
- **State = the raw `icon_url` string** (e.g. `"FullSun"`). Passthrough is the
  robust choice: it is always correct and surfaces any new value rather than
  hiding or rejecting it.
- **No `device_class`** — deliberately *not* `SensorDeviceClass.ENUM`, because
  ENUM requires a complete `options` list and would log "invalid state" for any
  value outside it. With the enumeration unknown, ENUM would be fragile.
- **No `state_class`** (it's a categorical string, not a measurement).
- **`icon` property** maps known values to MDI icons, with a neutral fallback for
  the unknown:
  - `FullSun` → `mdi:weather-sunny`
  - `LightRain` → `mdi:weather-rainy`
  - anything else → `mdi:help` (a question mark, signalling "unmapped value")
  The map is intentionally small (only what we've observed); it extends as new
  values appear, and an unmapped value still gets a correct *state*, just the
  question-mark icon.
- `native_value` → `self._day_data.icon_url` (guarded; `None` → unknown).
- English names: day 0 → "Weather condition"; day k → "Weather condition day +{k}".

**`async_setup_entry`** adds a condition sensor alongside the temperature and
rain sensors, one of each per configured day.

### Alternative considered (not chosen)

Mapping the state itself to Home Assistant weather-condition strings
(`sunny`/`rainy`/…) would be "friendlier", but every unmapped value would fall
back to a guess or `unknown`, losing information. Raw passthrough + an MDI icon
gives the visual nicety without the brittleness. Flag if you'd prefer the mapped
states instead.

## Out of scope

- Business hours and suggestion sensors — #6, #7.
- Entity name translations (nl/de) — deferred polish.

## Testing

- **Unit:** condition sensor for day 0 over `{0: iconUrl "FullSun"}` →
  `native_value == "FullSun"`, `icon == "mdi:weather-sunny"`,
  `unique_id` ends `_condition_0`, `device_class` and `state_class` are `None`.
- **Unit:** `LightRain` → `mdi:weather-rainy`; an unknown value (e.g. `"Blizzard"`)
  → `mdi:help`.
- **Unit:** missing day → `native_value` None, `icon` None, unavailable.
- **Integration (real HA core + `mock_api`):** after setup (forecast_days=3), the
  registry has 3 condition entities, and today reads state `"FullSun"`.
