# Issue #3 — Temperature sensor

Branch: `feature/3_temperature-sensor`

## Problem

Expose the expected temperature per forecast day as Home Assistant sensors.
This is the first concrete entity built on the #2 foundation, so it's also what
makes the "Burgers Zoo" device actually appear in HA.

Requirements (from the ticket):
- One sensor per configured forecast day (day 0 = today, up to `forecast_days`).
- `SensorDeviceClass.TEMPERATURE`, native unit °C, `SensorStateClass.MEASUREMENT`.
- Value from `DayData.temperature`; `None` → unknown.

## Direction

**New `BurgersZooTemperatureSensor(BurgersZooBaseEntity)`** in `sensor.py`:
- `key = "temperature"` → unique id `{entry_id}_temperature_{day}` (from the base).
- `_attr_device_class = SensorDeviceClass.TEMPERATURE`
- `_attr_native_unit_of_measurement = UnitOfTemperature.CELSIUS`
- `_attr_state_class = SensorStateClass.MEASUREMENT` (enables long-term statistics)
- `native_value` → `self.coordinator.data[self._day].temperature` (guarded; `None` → unknown).
- Availability is inherited from the base (unavailable when the day is missing).

**`async_setup_entry`** (currently adds nothing) creates one sensor per configured day:
```python
coordinator = entry.runtime_data
async_add_entities(
    BurgersZooTemperatureSensor(coordinator, entry, day)
    for day in range(coordinator.forecast_days)
)
```
Iterating `range(coordinator.forecast_days)` (not the fetched-data keys) keeps the
entity set stable even when a single day's fetch fails — those entities just go
unavailable rather than disappearing.

### Naming (the one open decision)

All per-day temperature entities sit under the one "Burgers Zoo" device, so the
day must be in the entity name. Proposed (via `_attr_name`, `has_entity_name=True`):
- day 0 → **"Temperature"** (friendly name "Burgers Zoo Temperature", entity `sensor.burgers_zoo_temperature`)
- day k>0 → **"Temperature day +{k}"** (e.g. `sensor.burgers_zoo_temperature_day_1`)

I'm deferring HA entity *translations* (nl/de names) to a later polish pass to
keep this ticket focused — names are English for now. Flag if you'd rather do
translation keys now, or prefer different day labels (e.g. "today/tomorrow").

## Out of scope

- The other sensors (rain, icon, business hours, suggestion) — #4–#7.
- Entity name translations (nl/de) — later polish.

## Testing

- **Unit (direct instantiation):** a `BurgersZooTemperatureSensor` for day 0 over a
  coordinator holding `{0: temp 30}` → `native_value == 30`, `device_class` temperature,
  unit `°C`, `state_class` measurement, `unique_id` ends with `_temperature_0`.
- **Unit:** day with no data → `native_value` is `None` and the entity is unavailable.
- **Integration (real HA core + `mock_api`):** set up the entry (forecast_days=3) and
  assert via the entity registry that 3 temperature entities are created for the
  config entry, and today's entity has state `"30"` with the temperature device class.
