# Issue #2 — Sensor platform foundation and device

Branch: `feature/2_sensor-platform-foundation`

## Problem

The merged base wires up the API client, coordinator, and config flow but
creates **no entities**. Before any concrete sensor (#3–#7) can be built, we need
the shared scaffolding: the sensor platform registered, a single device that all
entities group under, and a base entity class encapsulating what every sensor
shares. This ticket delivers that foundation and nothing user-visible beyond the
device wiring.

## Direction

**1. Register the sensor platform** (`__init__.py`)
- Change `PLATFORMS` from `[]` to `[Platform.SENSOR]`. Setup already forwards via
  `async_forward_entry_setups`, so no other change is needed.

**2. New `sensor.py`**
- `async_setup_entry(hass, entry, async_add_entities)` reads the coordinator from
  `entry.runtime_data`. It adds **no entities yet** — the per-day sensors are
  added by #3+. This file is the platform entry point they will extend.

**3. Base entity** — `BurgersZooBaseEntity(CoordinatorEntity[BurgersZooDataUpdateCoordinator])`
- `_attr_has_entity_name = True`.
- Constructor `(coordinator, entry, day, key)`:
  - stores `self._day`,
  - sets `_attr_unique_id = f"{entry.entry_id}_{key}_{day}"` (stable per sensor type per day),
  - sets `_attr_device_info` grouping all entities under one device:
    `identifiers={(DOMAIN, entry.entry_id)}`, `name=NAME`,
    `manufacturer="Burgers' Zoo"`, `entry_type=DeviceEntryType.SERVICE`,
    `configuration_url="https://www.burgerszoo.nl"`.
- `available` property: `super().available and self._day in self.coordinator.data`
  (an entity for a day the API didn't return becomes unavailable rather than erroring).
- Concrete sensors (#3+) subclass this and provide `native_value`, a
  `translation_key`/name, device class, and unit.

The `key` + `day` design is deliberate: every planned sensor is per-day with a
type key, so encoding that in the base keeps the concrete sensors tiny and DRY.

## Out of scope

- Any concrete sensor entity (temperature, rain, icon, business hours, suggestion) — those are #3–#7.
- Entity translation strings (added per sensor by the concrete tickets).

## Testing

Follows the existing pytest + pytest-homeassistant-custom-component + aioresponses setup.

- **Platform loads:** with `mock_api`, the config entry reaches `LOADED` with
  `Platform.SENSOR` in `PLATFORMS`, and unload still succeeds.
- **Base entity behaviour** (via a minimal test-only subclass, since the base is
  abstract until #3): asserts `device_info` identifiers/name/manufacturer/entry_type,
  the `unique_id` pattern `f"{entry_id}_{key}_{day}"`, `has_entity_name is True`,
  and that `available` is `False` when the day is absent from `coordinator.data`
  and `True` when present.
